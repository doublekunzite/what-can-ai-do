1. Design Philosophy (Minimalist & Bare Bones)

Visual Style:

  • Single-column layout with generous white space
  • Monochromatic color scheme (black text, white background, one accent color like #0066cc)
  • System fonts only (-apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif)
  • No images unless absolutely necessary
  • No JavaScript frameworks—vanilla JS only where essential

Core CSS Template:

* { margin:0; padding:0; box-sizing:border-box; }
body { 
  max-width: 700px; 
  margin: 40px auto; 
  padding: 0 20px; 
  font-family: -apple-system, sans-serif;
  line-height: 1.6;
  color: #222;
}
h1, h2 { margin: 1.5em 0 0.5em; }
a { color: #0066cc; text-decoration: none; }
a:hover { text-decoration: underline; }

2. Page-by-Page Content Strategy

Homepage (index.html)

Simple hero with clear navigation:

What can I do with AI?

A practical guide to understanding and using artificial intelligence.

[Explore Uses] [Learn Risks] [Compare Models] [Address Concerns]

Include a 2-sentence description of each section so visitors know where to go.

Use Cases Page (use-cases.html)

Organize by domains, not just technical capabilities.

Categories:

  • Creative (writing, art, music, video)
  • Analytical (data analysis, research, coding)
  • Communication (translation, summarization, email)
  • Productivity (task automation, scheduling, note-taking)
  • Learning (tutoring, explaining concepts, practice problems)
  • Professional (resume review, mock interviews, brainstorming)

Format for each use case:

### Use Case: Summarize Long Documents
What it does: Extracts key points from articles, papers, or reports
Examples:
- Turn a 20-page research paper into 5 bullet points
- Create meeting notes from a transcript
- Digest a long email thread

Try this prompt: "Summarize this article in 3 paragraphs: [paste text]"

Key feature: Make it interactive—include sample prompts people can copy and try immediately.

Risks Page (risks.html)

Balance honesty with nuance. Organize by severity and mitigation.

Sections:

  1. Privacy & Data (what happens to your inputs)
  2. Accuracy & Hallucinations (when AI confidently lies)
  3. Bias & Fairness (inherent biases in training data)
  4. Dependency & Skill Loss (over-reliance on automation)
  5. Environmental Impact (energy consumption)
  6. Economic Disruption (job transformation)

Format for each risk:

### Risk: AI "Hallucinations"

What it is: AI can generate plausible-sounding but false information.

Real example: A lawyer used ChatGPT for case research; it invented fake cases.

How to mitigate:
- Always verify factual claims
- Use AI for drafts, not final authority
- Cross-check with primary sources

Severity: ⚠️⚠️⚠️ (High)

Add a "Risk Mitigation Checklist" at the end—a simple list of habits users should develop.

LLM Comparison Page (compare.html)

Keep it simple and actionable. Compare 4-5 major models.

Comparison Table (responsive, minimal):

Model      | Best For        | Cost   | Speed | Accuracy | Context Window
-----------|-----------------|--------|-------|----------|---------------
GPT-4o     | Complex reasoning| $$     | Fast  | High     | 128K tokens

  • "I need to analyze a 100-page PDF" → Claude
  • "I want something free and private" → Llama

"Isn't AI Bad?" Page (faq.html)

Frame this as a dialogue. List common objections and give nuanced, non-defensive responses.

Structure:

Concern: "AI is just stealing from artists/writers."

Response:
- Acknowledge the concern: "This is a valid and complex issue."
- Explain the technical reality: Training data vs. copying
- Discuss current mitigation: Opt-out systems, licensing deals
- Present both sides: "Some argue... Others counter..."
- Your action: "If this worries you, consider using models that train on licensed data."

Related: [Link to risks page section on copyright]

Include 6-8 common concerns:

  1. "It'll take all our jobs"
  2. "It's just autocomplete, not real intelligence"
  3. "It's environmentally destructive"
  4. "It's dumbing people down"
  5. "It's biased and discriminatory"
  6. "It's a hype bubble"

End with: "Your concerns are valid. AI is a tool, not a religion. Use it where it helps, question it where it harms."

4. Content Best Practices

Tone:

  • Plain language, no jargon
  • Acknowledge uncertainty ("We don't know yet...")
  • Action-oriented (always include "Try this:" or "Here's what to do:")

Maintainability:

  • Write content in Markdown first, then convert to HTML
  • Keep a content.md file with all text for easy updates
  • Date-stamp your comparisons: "Updated: November 2024"

Accessibility:

  • Semantic HTML only (<h2>, <p>, <ul>)
  • Ensure 4.5:1 contrast ratio
  • Works without JavaScript

Launch with just 3 pages if needed: Homepage, Use Cases, and FAQ. Add the others later.

Here’s what to do with the HTML files, ordered from simplest to most technical:

Option 1: GitHub Pages (Free, Best for This Project)

This is ideal for a static, minimalist site. No servers, databases, or WordPress complexity.

Steps:

  1. Create a GitHub account (github.com)
  2. Install GitHub Desktop (desktop.github.com) - visual interface, no command line needed
  3. Create a new repository named what-can-ai-do
  4. Drag your HTML/CSS files into the GitHub Desktop app and click "Commit to main"
  5. Go to repository Settings > Pages and set source to "main branch"
  6. Your site is live at username.github.io/what-can-ai-do

Pros:

  • Completely free
  • Ultra-fast loading
  • No maintenance or security updates needed
  • Perfect for static HTML/CSS

Cons:

  • Small learning curve if you've never used Git

Option 2: Traditional Web Hosting (Familiar cPanel)

If you want something closer to WordPress workflow but simpler.

Steps:

  1. Buy hosting from providers like:
    • NearlyFreeSpeech.net ($5/year for tiny sites)
    • Porkbun (cheap hosting + domain bundles)
    • Any host with cPanel
  2. Skip the WordPress installer - look for "File Manager" in cPanel
  3. Upload your files:
    • Open File Manager
    • Navigate to public_html folder
    • Upload your 5 HTML files and CSS folder
  4. Your homepage will be index.html automatically

Pros:

  • Feels familiar if you've used hosting before
  • Can add email, domain easily
  • No command line

Cons:

  • Costs $20-50/year
  • Slight overkill for a static site

Option 3: Netlify Drop (Easiest Drag-and-Drop)

The absolute simplest method.

Steps:

  1. Go to app.netlify.com/drop
  2. Drag your entire folder of HTML/CSS files onto the page
  3. Done. Your site is live with a random URL
  4. Optional: Connect a custom domain later

Pros:

  • No account setup needed initially
  • Instant deployment
  • Free HTTPS certificate
  • Super fast

Cons:

  • Less control than GitHub Pages
  • Can feel "magical" in an opaque way

Option 4: WordPress (If You Must)

You can use WordPress, but it defeats the minimalist purpose. Here's how:

  1. Create a "Static Page" for each of your 5 pages
  2. Switch to a minimalist theme like "Twenty Twenty-Four" or "Astra"
  3. Disable comments globally
  4. Edit each page in "Code Editor" mode (not Visual Editor)
  5. Paste your HTML into the code view
  6. Disable plugins you don't need for speed

Warning: This adds complexity, security updates, and loading time for no benefit on a static site.

How to Edit the Files

You'll need a text editor better than Notepad:

  • Windows: Notepad++ or VS Code
  • Mac: VS Code or TextMate
  • Cross-platform: VS Code (free, best choice)

To edit:

  1. Open the .html file in the editor
  2. Make changes
  3. Save
  4. Re-upload to your host (or sync via GitHub Desktop)

Domain Name Setup

Once your site is live at a temporary URL:

  1. Buy a domain from Namecheap, Porkbun, or Cloudflare ($10-15/year)
  2. In your host's settings, add the domain
  3. Update DNS records (your host will give you exact values):
    • Point @ and www to their servers

For GitHub Pages, you just add a CNAME file with your domain name.

My Recommendation

Start with Netlify Drop to test everything. If you like it, move to GitHub Pages for long-term control. Both are free and perfectly match your minimalist vision.

WordPress is the wrong tool for a 5-page static educational site—it adds 20+ database tables and PHP processing for no benefit.

User: before you go ahead and draft the full HTML for the main page, a friend expressed security concerns about using AI to code. any thoughts or advice on avoiding potential risks? can you adjust the code to ensure security?

Assistant:

Excellent question. Your friend's concerns are valid, and it's smart to address this before building. Here are the specific risks and how to mitigate them:

AI Code Generation Risks (And Why They're Minimal Here)

Real Risks:

  1. Hidden malicious code - AI could insert invisible tracking scripts or crypto-miners
  2. Outdated dependencies - Recommending libraries with known vulnerabilities
  3. Insecure patterns - Improper input sanitization (if you add forms later)
  4. CDN hijacking - Linking to external scripts that could be compromised

Why Your Static Site is Inherently Secure:

  • No server-side code → No database injections, no server hacking
  • No user logins → No authentication bypasses
  • No dynamic content → No XSS attacks (unless you add forms)
  • No external dependencies → No supply chain attacks

Security-First Code Principles

Here's how I'll adjust the code:

✅ DO:

  • Zero external dependencies - No CDN links, no Google Fonts, no third-party scripts
  • Self-contained everything - All CSS, all content in one folder
  • Semantic HTML - Prevents certain parsing attacks
  • Minimal JavaScript - Only vanilla JS, and only if essential
  • Clear comments - So you can audit every line

❌ DON'T:

  • No <script src="https://..."> tags
  • No @import url('https://...') in CSS
  • No embedded iframes or external content
  • No analytics scripts (unless you add them yourself later)
  • No forms (for now) - they require input validation

Secure File Structure

/what-can-ai-do/
├── index.html
├── use-cases.html
├── risks.html
├── compare.html
├── faq.html
├── css/
│   └── style.css

    └── main.js (optional - only if you want copy-to-clipboard)

No other files. No node_modules. No package.json. No build process.

Manual Security Audit Checklist

Before uploading, manually verify every line:

  • No <script> tags pointing to external URLs
  • No <link> tags to external stylesheets
  • No <iframe> elements
  • No onload="..." or inline JavaScript events
  • No hidden elements (display: none; should be suspicious)
  • No base64-encoded blobs you don't understand
  • All text content makes sense when you read it

If you see something you don't understand, delete it and ask me to explain.
